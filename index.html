Project Goals

The objectives of the project changed Sprint to Sprint, especially early on. Since we were originally tasked with creating an app, our goals and methods evolved completely by Sprint 2.

Sprint 1 Goals

The initial goals from the beginning were to ensure that we became familiar enough with our tools to proceed efficiently. This involved researching Unity and other potential libraries/languages that would allow us to reach our final deliverable.
  1. Become Familiar with Unity
  2. Create Clickable Prototype for the Program
  3. Begin Creating the Quiz Structure
  4. Start to Create Menus

Sprint 2 Goals

Upon shifting our focus formally to Babylon.js, we aimed to roughly demonstrate what our final project would be capable of doing. This included creating all the individual components to prove functionality.
  1. Create Formal Menus
  2. Begin 2D Modeling in Unity (later changed to Babylon.js)
  3. Start Creating Player Architectures
  4. Start Creating Basic Play Components (i.e. routes, schemes)
  5. Explore using Babylon.js for the Project
  6. Introduce/Translate to a 3D Model

Sprint 3 Goals

At this stage, we began to organize our efforts into a formal program. In essence, we worked to tie our components into one working system, and then deploy it. We also worked to ensure that any problematic issues or bugs were handled properly.
  1. Continue to Research Babylon.js as a Development Option
  2. Finalize Route Tree Implementation
  3. Ensure Routes are available to each Player on the screen
  4. Connect Menus with the Simulation Screens
  5. Begin 3D Modeling
  6. Deploy to Website

Project Approach

Originally, our group was presented with creating an app that would connect coaches with athletes. We were given a clickable prototype of the app and tasked with providing functionality to it. However, soon after we shifted focus towards development of a quizzing platform.

The new objective was to create a complete program that allows coaches to quiz football players on specific routes and formations. In addition, the program needed to be able to generate animations for the plays in 2D and 3D, allowing the user to view the play from multiple viewpoints and angles.

Initially, we started programming with the Unity platform. This entailed writing scripts in C#, alongside using the graphical interface provided by Unity to produce the animations. However, a series of problems emerged with this approach. Primarily, Unity, although powerful, had a significant learning curve to get our project to a functional state. Routing the sprites, interacting between menus, and quick loading led us to seek a new approach.

After voicing some of our concerns, we were introduced to BabylonJS, a JavaScript framework that had the potential to fix many prior issues. From here, we began to develop in strictly BabylonJS, converting the already existing Unity project into JavaScript. The built in functionality and quick testing made this conversion successful.

Lastly, we finalized the program by using BabylonJS, saved HTML pages that integrated the JavaScript code into a quickly loadable format, and deployed the project to GitHub's hosting service.

Tools, APIs, and Frameworks Used

Unity

The initial approach was to use Unity to develop our entire platform. However, we determined that the speed, learning curve, and practicality of continuing to rely on Unity would prove to be a hinderance.

BabylonJS

Our final, primary approach was to use the JavaScript platform "BabylonJS", which allowed us to tie together our Quiz, 2D, and 3D functionality into one coherant program. With BabylonJS, we were able to experiment in their provided playground (playground.babylonjs.com). From the playground, we downloaded the playground into individual HTML files, which we finally modified to allow them to pass data and interact via Session Storage.

GitHub

Used for coordination, version management, and hosting of program site.

Project Experience

(STEP 1) Main Menu: The user chooses from a list of various routes for each receiver to run. From there, they may choose to see this play out in either a 2d or 3d simulation.
(STEP 2a) 2D Simulation: The user can see the chosen routes being run by each receiver in a 2d environment. From there the user can access the quiz to test their knowledge of which receiver ran which route.
(STEP 2b) 3D Simulation: The user can watch the players run the routes on a loop in a 3d environment. They can choose to see it play out from each player's perspective, either on offense or defense. From there the user can access the quiz to test their knowledge of which receiver ran which route
(STEP 3) Quiz Screen: The user is presented a question about which route a receiver ran and multiple choice answers. If they choose incorrectly, the answer turns red and they can choose another answer. If they choose correctly, an alert tells them so and the answer turns green. From there, the user is either prompted to answer a second question or is presented a button to loop back to the main menu.
